# E2E 테스트 지침

---

## 1. 개요

Frontend ↔ Backend 간 전체 사용자 워크플로우를 브라우저 환경에서 검증하는 E2E 테스트 지침입니다.
프로덕션 환경(nerdctl compose)에서 실제 서비스를 기동한 후 브라우저 기반 테스트를 수행합니다.

---

## 2. 테스트 프레임워크 권장

| 항목 | 권장 도구 |
|------|-----------|
| E2E 프레임워크 | Playwright 또는 Cypress |
| 브라우저 | Chromium (Playwright 내장) |
| 실행 방식 | headless (CI) / headed (로컬 디버깅) |

### 향후 설치 (Playwright 기준)
```bash
cd packages/frontend
npm install -D @playwright/test
npx playwright install
```

---

## 3. 테스트 환경 준비

### 3.1 프로덕션 환경 시작
```bash
# 전체 서비스 시작
nerdctl compose up -d --build

# 서비스 상태 확인
nerdctl compose ps

# Frontend 접속 확인
curl -s -o /dev/null -w "%{http_code}" http://localhost/
# 기대: 200
```

### 3.2 테스트 데이터 초기화
```bash
# 볼륨 초기화 후 재시작 (깨끗한 상태)
nerdctl compose down -v
nerdctl compose up -d --build
```

## 4. E2E 테스트 시나리오

### 시나리오 E1: 관리자 로그인 → 대시보드 접근

**목적**: 관리자 로그인 UI → JWT 저장 → 대시보드 페이지 접근 검증

**테스트 순서**:
1. http://localhost/admin/login 접속
2. storeIdentifier: "default", username: "admin", password: "admin123" 입력
3. 로그인 버튼 클릭
4. /admin/dashboard 리다이렉트 확인
5. 대시보드 페이지 렌더링 확인

**검증 항목**:
- [ ] 로그인 폼 렌더링 (storeIdentifier, username, password 필드)
- [ ] 정상 로그인 시 /admin/dashboard 이동
- [ ] localStorage에 admin_token 저장
- [ ] 잘못된 비밀번호 시 에러 토스트 표시
- [ ] 5회 실패 시 계정 잠금 안내 메시지

---

### 시나리오 E2: 관리자 메뉴/카테고리 관리

**목적**: 관리자 UI에서 카테고리 생성 → 메뉴 등록 → 수정 → 삭제 흐름 검증

**사전 조건**: 시나리오 E1에서 관리자 로그인 완료

**테스트 순서**:
1. /admin/menus 페이지 접속
2. 카테고리 추가 ("음료")
3. 메뉴 추가 (이름: "아메리카노", 가격: 4500, 카테고리: "음료")
4. 메뉴 목록에서 등록된 메뉴 확인
5. 메뉴 수정 (가격 → 5000)
6. 메뉴 삭제
7. 카테고리 삭제

**검증 항목**:
- [ ] 카테고리 추가 후 목록에 표시
- [ ] 미분류 카테고리 삭제 버튼 비활성화 또는 에러
- [ ] 메뉴 추가 폼 (이름, 가격, 설명, 카테고리, 이미지)
- [ ] 메뉴 등록 후 목록에 표시
- [ ] 메뉴 수정 후 변경 사항 반영
- [ ] 메뉴 삭제 확인 다이얼로그 → 삭제 후 목록에서 제거
- [ ] 드래그 앤 드롭으로 메뉴/카테고리 순서 변경

---

### 시나리오 E3: 관리자 테이블 설정 → 고객 메뉴 탐색

**목적**: 테이블 설정 → 고객 태블릿 인증 → 메뉴 탐색 흐름 검증

**테스트 순서**:
1. /admin/tables 페이지에서 테이블 1번 설정 (비밀번호: table1234)
2. 새 브라우저 탭에서 고객 인증 (테이블번호: 1, 비밀번호: table1234)
3. /customer/menu 페이지 접속
4. 카테고리 사이드바에서 카테고리 선택
5. 메뉴 카드 클릭 → 상세 모달 확인

**검증 항목**:
- [ ] 테이블 설정 폼 (테이블 번호, 비밀번호)
- [ ] 설정 후 테이블 목록에 active 상태 표시
- [ ] 고객 인증 후 /customer/menu 이동
- [ ] 카테고리 사이드바 렌더링 (미분류 포함)
- [ ] 메뉴 카드 (이름, 가격, 이미지) 표시
- [ ] 메뉴 상세 모달 (설명, 수량 선택, 장바구니 추가 버튼)

---

### 시나리오 E4: 고객 주문 → 관리자 실시간 수신

**목적**: 고객 장바구니 → 주문 제출 → 관리자 대시보드 실시간 반영 검증

**사전 조건**: 시나리오 E3에서 메뉴 등록 + 테이블 설정 + 고객 인증 완료

**테스트 순서**:
1. 고객: 메뉴 상세에서 수량 2 선택 → 장바구니 추가
2. 고객: 하단 장바구니 바 클릭 → /customer/cart 이동
3. 고객: 장바구니 항목 확인 → 주문하기 클릭
4. 고객: /customer/order/confirm 에서 주문 확인 → 주문 제출
5. 고객: /customer/order/success 페이지 표시 (주문번호, 카운트다운)
6. 관리자: /admin/dashboard에서 테이블 카드에 신규 주문 표시 확인 (SSE)

**검증 항목**:
- [ ] 장바구니 추가 시 하단 바에 수량/총액 업데이트
- [ ] 장바구니 페이지에서 수량 변경/삭제 동작
- [ ] 주문 확인 페이지에서 항목/총액 표시
- [ ] 주문 제출 후 성공 페이지 이동 (주문번호 표시)
- [ ] 관리자 대시보드에 실시간 신규 주문 알림 (토스트 + 카드 하이라이트)
- [ ] 고객 주문 이력 페이지에서 주문 표시

---

### 시나리오 E5: 관리자 주문 상태 변경 → 고객 실시간 수신

**목적**: 관리자 주문 상태 변경 → 고객 화면 실시간 반영 검증

**사전 조건**: 시나리오 E4에서 주문 생성 완료

**테스트 순서**:
1. 관리자: 대시보드에서 테이블 카드 클릭 → 주문 목록 패널
2. 관리자: 주문 상태 pending → preparing 변경
3. 고객: /customer/orders 에서 주문 상태 "조리중" 표시 확인 (SSE)
4. 관리자: 주문 상태 preparing → completed 변경
5. 고객: 주문 상태 "완료" 표시 확인

**검증 항목**:
- [ ] 관리자 주문 상태 변경 UI (버튼/드롭다운)
- [ ] 상태 변경 후 관리자 화면 즉시 반영
- [ ] 고객 화면에 SSE로 상태 변경 실시간 반영
- [ ] completed 상태에서 추가 변경 불가

---

### 시나리오 E6: 이용 완료 → 과거 내역 조회

**목적**: 관리자 이용 완료 처리 → 고객 세션 종료 → 과거 내역 조회 검증

**사전 조건**: 시나리오 E5에서 주문 완료 상태

**테스트 순서**:
1. 관리자: /admin/tables에서 테이블 1번 이용 완료 클릭
2. 관리자: 확인 다이얼로그 → 확인
3. 관리자: 테이블 상태 completed 확인
4. 고객: 세션 종료 안내 표시 확인
5. 관리자: /admin/history에서 과거 주문 내역 조회

**검증 항목**:
- [ ] 이용 완료 확인 다이얼로그 표시
- [ ] 이용 완료 후 테이블 상태 변경 (completed)
- [ ] 고객 화면에 세션 종료 안내
- [ ] 과거 주문 내역 페이지에서 아카이브된 주문 표시
- [ ] 날짜 필터 동작
