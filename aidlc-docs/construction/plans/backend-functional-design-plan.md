# Backend Functional Design Plan - 테이블오더 서비스

---

## 계획 개요

Backend 유닛의 상세 비즈니스 로직, 도메인 엔티티, 비즈니스 규칙을 설계합니다.
Application Design에서 정의된 6개 모듈(Auth, Menu, Order, Table, Realtime, System)의 내부 동작을 구체화합니다.

---

## Part 1: 질문

### Q1. 주문 상태 전이 규칙
주문 상태 변경 시 허용되는 전이 경로를 결정합니다.

A) 단방향만 허용 — 대기중 → 준비중 → 완료 (역방향 불가)
B) 유연한 전이 — 대기중 ↔ 준비중 ↔ 완료 (역방향 허용)
C) 단방향 + 취소 — 대기중 → 준비중 → 완료, 어느 단계에서든 "취소" 가능
D) 기타 (직접 입력)

[Answer]: B

### Q2. 테이블 세션 만료 처리
16시간 세션 만료 시 처리 방식을 결정합니다.

A) 자동 이용 완료 — 만료 시 자동으로 주문 이력 이동 및 리셋
B) 만료 상태 표시 — 만료 표시만 하고, 관리자가 수동으로 이용 완료 처리
C) 자동 갱신 — 활성 주문이 있으면 세션 자동 연장
D) 기타 (직접 입력)

[Answer]: B

### Q3. 주문 삭제 시 데이터 처리
주문 삭제 시 데이터를 어떻게 처리할지 결정합니다.

A) 물리 삭제 — DB에서 완전 삭제 (order_items 포함)
B) 논리 삭제 — deleted 플래그 설정, 조회 시 제외
C) 이력 이동 — 삭제된 주문을 별도 삭제 이력 테이블로 이동
D) 기타 (직접 입력)

[Answer]: B

### Q4. 카테고리 삭제 시 메뉴 처리
카테고리 삭제 시 해당 카테고리에 속한 메뉴의 처리 방식을 결정합니다.

A) 삭제 차단 — 메뉴가 있는 카테고리는 삭제 불가 (먼저 메뉴 이동/삭제 필요)
B) 미분류 이동 — 해당 메뉴를 "미분류" 카테고리로 자동 이동
C) 함께 삭제 — 카테고리와 소속 메뉴 모두 삭제 (확인 팝업)
D) 기타 (직접 입력)

[Answer]: B

### Q5. 이미지 파일 관리 정책
메뉴 삭제/수정 시 이미지 파일 처리 방식을 결정합니다.

A) 즉시 삭제 — 메뉴 삭제 시 이미지 파일도 즉시 삭제
B) 유지 — 이미지 파일은 삭제하지 않고 유지 (디스크 관리는 수동)
C) 교체 시 삭제 — 이미지 수정 시 이전 파일 삭제, 메뉴 삭제 시에도 삭제
D) 기타 (직접 입력)

[Answer]: A

### Q6. 이용 완료 후 주문 이력 데이터 구조
과거 주문 이력(order_history)에 저장할 데이터 형태를 결정합니다.

A) 정규화 유지 — orders/order_items 구조 그대로 이력 테이블에 복사
B) 비정규화 스냅샷 — 주문 전체를 JSON으로 직렬화하여 단일 레코드로 저장
C) 요약 저장 — 주문 요약 정보(총액, 메뉴 수, 시각)만 저장
D) 기타 (직접 입력)

[Answer]: B

### Q7. 관리자 초기 계정 정보 설정
시스템 초기화 시 생성되는 관리자 계정의 기본값 설정 방식을 결정합니다.

A) 환경 변수 — Docker 환경 변수로 매장 식별자/사용자명/비밀번호 설정
B) 하드코딩 기본값 — 코드에 기본값 설정 (admin/admin123 등), 첫 로그인 시 변경 권장
C) 설정 파일 — config 파일에서 읽어오기
D) 기타 (직접 입력)

[Answer]: B

### Q8. 메뉴 노출 순서 관리 방식
메뉴 노출 순서를 어떻게 관리할지 결정합니다.

A) 정수 기반 — display_order 정수 필드, 드래그앤드롭 시 순서 재계산
B) 간격 기반 — display_order에 100 단위 간격 부여 (100, 200, 300...), 중간 삽입 용이
C) 생성 순서 — 별도 순서 관리 없이 생성 시간 순 표시
D) 기타 (직접 입력)

[Answer]: B

### Q9. 테이블 세션 이용 완료 후 주문 데이터 처리 범위
이용 완료 시 orders/order_items 테이블의 데이터를 어떻게 처리할지 결정합니다.

A) 이동 후 삭제 — order_history로 이동 후 원본 orders/order_items에서 삭제
B) 복사 후 유지 — order_history로 복사하고 원본은 유지 (플래그로 구분)
C) 이동만 — order_history로 이동, 원본 삭제 (Q6 답변에 따라 형태 결정)
D) 기타 (직접 입력)

[Answer]: B

### Q10. 동시 주문 처리 전략
같은 테이블에서 동시에 여러 주문이 들어올 때의 처리 방식을 결정합니다.

A) 순차 처리 — 각 주문을 독립적으로 순차 생성 (별도 주문 번호)
B) 병합 처리 — 짧은 시간 내 동일 테이블 주문은 하나로 병합
C) 낙관적 동시성 — 각 주문 독립 생성, 총액은 DB 레벨에서 원자적 계산
D) 기타 (직접 입력)

[Answer]: A
