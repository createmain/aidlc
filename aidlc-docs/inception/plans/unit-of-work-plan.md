# Units Generation Plan - 테이블오더 서비스

---

## 계획 개요

모노레포 구조의 테이블오더 MVP를 개발 가능한 유닛으로 분해합니다.
Application Design에서 정의된 6개 백엔드 모듈, 2개 프론트엔드 영역, DB 스키마를 기반으로 유닛 경계와 구현 순서를 결정합니다.

---

## Part 1: 질문

### Q1. 유닛 분해 전략 (Story Grouping)
모노레포 내에서 유닛을 어떻게 나눌지 결정합니다.

- A: **계층별 분해** — DB 스키마 → 백엔드 API → 프론트엔드 순서로 3개 유닛
- B: **도메인별 분해** — Auth, Menu, Order, Table 등 도메인별로 유닛 분리 (각 유닛이 백엔드+프론트엔드 포함)
- C: **배포 단위별 분해** — Backend(API 서버), Frontend(React 앱), Database(스키마/마이그레이션) 3개 유닛
- D: 기타 (직접 입력)

[Answer]: C

### Q2. DB 스키마 유닛 분리
DB 스키마와 마이그레이션을 별도 유닛으로 분리할지 결정합니다.

- A: **별도 유닛** — DB 스키마를 독립 유닛으로 분리하여 먼저 구현
- B: **백엔드에 포함** — DB 스키마를 백엔드 유닛 내에서 함께 구현

[Answer]: B

### Q3. 프론트엔드 유닛 세분화
단일 React 앱 내 고객 UI와 관리자 UI를 어떻게 유닛으로 다룰지 결정합니다.

- A: **단일 유닛** — 고객 UI + 관리자 UI를 하나의 프론트엔드 유닛으로
- B: **분리 유닛** — 고객 UI와 관리자 UI를 별도 유닛으로 (같은 React 앱이지만 개발 단위 분리)

[Answer]: A

### Q4. Realtime 모듈 유닛 배치
Realtime(SSE) 모듈을 어떤 유닛에 포함시킬지 결정합니다.

- A: **백엔드 유닛에 포함** — Order/Table 모듈과 함께 백엔드 유닛 내에서 구현
- B: **별도 유닛** — Realtime을 독립 유닛으로 분리하여 별도 구현

[Answer]: A

### Q5. 구현 순서 우선순위
유닛 간 구현 순서를 결정합니다.

- A: **의존성 기반** — DB → 백엔드 핵심(Auth, Menu) → 백엔드 주문(Order, Table, Realtime) → 프론트엔드 순서
- B: **기능 슬라이스** — 메뉴 조회 전체(DB+API+UI) → 주문 전체(DB+API+UI) → 관리 기능 순서
- C: **백엔드 우선** — 백엔드 전체 완성 → 프론트엔드 전체 구현
- D: 기타 (직접 입력)

[Answer]: C
